# StudyBuddy 英语课外班改造评估报告

> **评估日期**：2026 年 7 月 6 日  
> **评估对象**：AI 伴学系统 → 英语课外班教学管理平台  
> **核心问题**：在现有项目上增加班级功能，还是直接开发特化版本？

## 一、需求拆解

| 需求 | 复杂度 | 影响范围 |
|------|--------|----------|
| 全部使用 Kimi Code API | ⭐ 低 | 仅配置层（去掉 DeepSeek，统一模型） |
| 去除数学和对话功能 | ⭐⭐ 中 | 前后端各删 50%+ 代码，路由/导航/数据全面裁剪 |
| 班级管理：学生共用资料，不共用错题本 | ⭐⭐⭐⭐ 高 | 数据隔离模型从「单人单学科」变为「班级共享 + 个人私有」 |
| 管理员批量管理班级 | ⭐⭐⭐ 中高 | 权限体系从「单 root」变为「总管理员 + 班级管理员」 |
| 整合多人错题本 + 单人错题本 | ⭐⭐⭐ 中 | 需要新增聚合查询和数据展示层 |

## 二、现有架构核心特征

### 2.1 数据隔离模型（单一维度）

```
data/users/
├── {username}.json              # 用户资料
├── {username}_config.json        # 个人配置（API Key、模型、压缩等）
├── {username}_数学_problems.json  # 数学题库
├── {username}_英语_problems.json  # 英语题库
├── {username}_英语_materials.json # 英语资料
├── {username}_英语_wrong.json      # 英语错题本
└── ...
```

**关键特征**：所有数据文件以 `username` 为唯一前缀，学科作为后缀维度。没有「班级/群组」概念。

### 2.2 权限体系（单层）

- 普通用户：Cookie Session 认证，无管理员权限
- 管理员：唯一硬编码用户 `root`，通过 `username == "root"` 判断
- 管理员权限检查散布在 `admin.py` 的 `get_current_admin` 及多个端点中

### 2.3 学科系统（项目灵魂）

- 登录时必须选择「数学 / 英语 / 对话」
- 学科决定数据存储前缀（`{username}_{subject}_*.json`）
- 前端路由守卫根据学科跳转不同首页（数学 → /solve，英语 → /english-upload，对话 → /chat）
- 左侧导航栏根据学科动态显示不同入口
- 后端 18 个端点中，约 12 个包含学科相关逻辑

### 2.4 模型分流（双引擎）

- 数学：含图/PDF → Kimi，纯文字 → DeepSeek（flash/pro 可选）
- 英语：强制 DeepSeek
- 对话：用户可选 DeepSeek 或 Kimi

### 2.5 代码规模

| 模块 | 文件数 | 代码行数 | 备注 |
|------|--------|----------|------|
| 后端端点 | 18 个 | ~6,000 行 | 含数学/英语/对话/管理 |
| 前端视图 | 16 个 | ~10,000 行 | 含数学/英语/对话/管理 |
| 核心工具 | 4 个 | ~500 行 | ai_client、file_lock、pricing、paths |
| 总计 | — | ~18,500 行 | 不含测试和构建产物 |

### 3.3 核心痛点

#### 痛点 1：数据模型的兼容灾难

现有代码中，几乎每个涉及数据读取的地方都调用 `user_data_path(username, subject, suffix)`。引入班级后，资料需要按 `class_id` 存储，但错题本仍按 `user_id` 存储。这意味着：

- 资料端点需要同时支持「读取个人资料」和「读取班级资料」
- 所有 `user_data_path` 调用处需要增加条件判断：`if shared → class_path else → user_path`
- 上传逻辑需要区分「存入个人库」和「存入班级库」
- JSON 文件命名约定需要扩展：`{class_id}_materials.json` 与 `{username}_英语_materials.json` 并存

这种「补丁式」改造会导致代码中充满条件分支，可读性和维护性大幅下降。

#### 痛点 2：权限体系的硬编码扩散

现有管理员判断是 `username == "root"`，散布在：
- `admin.py:17`：`if username != "root"`
- `auth.py:86`：`if username == "root"`
- `auth.py:125`：`user["is_admin"] = (username == "root")`
- `frontend/stores/auth.js:85`：`isAdmin = computed(() => user.value?.username === "root")`
- `frontend/router/index.js:65`：`if (auth.isLoggedIn && auth.isAdmin)`

改为多层级权限（总管理员 + 班级管理员）后，需要：
- 用户数据中增加 `role` 字段（`student` / `class_admin` / `super_admin`）
- 每个受保护端点增加权限检查（学生不能访问班级管理，班级管理员只能管理自己的班级）
- 前端导航栏需要动态渲染（不同角色看到不同入口）
- 班级管理员的资料修改权限需要精确到班级级别

这相当于重写整个认证/授权层。

#### 痛点 3：学科系统的「幽灵残留」

即使删除了数学/对话的页面和端点，代码中仍有大量学科相关的条件分支：

```python
# 示例：散布在各处的学科判断
if subject == "数学":
    ...
elif subject == "英语":
    ...
else:  # 对话
    ...
```

这些分支在代码中散布了几十处，清理不干净会留下技术债务。而且现有的数据迁移逻辑（`_migrate_legacy_data`）和旧格式兼容代码也需要维护。

#### 痛点 4：前端路由守卫的彻底重写

现有路由守卫的核心逻辑是：
1. 检查登录状态
2. 检查学科匹配（`routeSubject !== auth.subject`）
3. 管理员跳转到 `/admin`

改为班级管理后，守卫逻辑变为：
1. 检查登录状态
2. 检查角色权限（学生/班级管理员/总管理员）
3. 检查班级成员身份（该学生是否属于目标班级）
4. 检查资源访问权限（该学生是否有权访问某份资料）

这是完全不同的权限模型，需要完全重写。

## 四、方案 B：开发特化版本（推荐）

### 4.1 设计思路

以「英语课外班教学管理」为原生目标，从零设计数据模型和权限体系。直接复制原项目的**核心工具层**，重写业务层。

### 4.2 架构设计

```
新项目的核心设计原则：
┌─────────────────────────────────────────────────────┐
│  数据层：班级共享 vs 个人私有，原生分离，无兼容层      │
│  权限层：总管理员 + 班级管理员 + 学生，三级角色原生    │
│  功能层：仅英语，无学科概念，无模型选择，强制 Kimi Code  │
│  前端层：精简导航，班级管理作为一等公民               │
└─────────────────────────────────────────────────────┘
```

#### 数据文件组织

```
data/
├── classes/
│   ├── {class_id}.json              # 班级信息（名称、创建时间、班级管理员）
│   ├── {class_id}_materials.json     # 班级共享资料库
│   ├── {class_id}_students.json      # 班级学生名单
│   └── {class_id}_wrong_aggregate.json # 班级错题聚合（定时生成/实时计算）
├── users/
│   ├── {username}.json               # 用户资料（姓名、班级、角色、年级）
│   ├── {username}_config.json        # 个人配置（Kimi Code API Key、压缩参数）
│   ├── {username}_wrong.json         # 个人错题本（私有）
│   ├── {username}_solve_sessions.json # 个人解题记录
│   └── {username}_usage.json         # 个人用量统计
└── uploads/
    └── ...                           # 上传文件（与原项目相同）
```

**关键设计决策**：
- 资料天然共享：上传文件直接关联 `class_id`，所有同班学生自动可见
- 错题天然私有：错题本始终绑定 `username`，无班级维度
- 错题聚合：通过后端端点实时汇总班级内所有学生的错题，不单独存储
- 配置集中管理：Kimi Code API Key 可以由总管理员统一配置，学生无需配置

| 从零开发时间不可控 | 低 | 核心模块复用，实际编码时间可控 |
| 缺乏旧代码参考 | 低 | 可随时对照原项目实现 |
| 测试覆盖不足 | 中 | 功能范围更小，测试更容易 |
| 数据迁移（如果需要） | 低 | 新项目不需要迁移旧数据 |

## 五、方案对比总结

| 维度 | 方案 A：改造现有项目 | 方案 B：开发特化版本 |
|------|----------------------|----------------------|
| **总工作量** | 30-40 小时 | 29-39 小时 |
| **后端改动文件** | 修改 15+，删除 6+，新增 4+ | 重写 10 个，复用 4 个工具 |
| **前端改动文件** | 删除 6+，修改 4+，新增 2+ | 重写 8 个，复用通用组件 |
| **数据模型** | 补丁式兼容（旧 + 新并存） | 原生设计（班级 + 个人） |
| **权限体系** | 从单层硬编码改多层动态 | 原生三级角色 |
| **代码质量** | 中等（有兼容层和历史包袱） | 高（纯净、无幽灵代码） |
| **长期维护** | 困难（条件分支多，理解成本高） | 容易（职责清晰，模块独立） |
| **Bug 风险** | 高（旧代码清理不干净） | 低（全新代码，测试覆盖） |
| **与原项目关系** | 耦合（原项目被改造） | 解耦（原项目保留独立） |
| **未来扩展** | 受限（兼容层限制） | 灵活（原生架构可扩展） |
| **数据迁移** | 需要（旧格式 → 新格式） | 不需要（全新数据） |
| **部署** | 共用（无法独立部署） | 独立（可与原项目并存） |

## 六、最终建议

### 强烈推荐：方案 B（开发特化版本）

**核心理由**：

1. **工作量相当，但质量天壤之别**。两个方案的总工时几乎相同，但方案 B 产出的是干净、可维护的代码，方案 A 产出的是充满兼容层和条件分支的「弗兰肯斯坦」代码。

2. **数据模型是根本差异**。现有项目的「单人单学科」模型与班级管理的「多人共享」模型在底层是冲突的。方案 A 是在错误的地基上打补丁，方案 B 是从正确的地基开始建造。

3. **权限体系是架构级变更**。从单 `root` 到三级角色，这不是「加个字段」能解决的，而是认证中间件、路由守卫、前端导航、API 端点的全面重构。与其在旧代码中到处打补丁，不如原生设计。

4. **保留原项目 = 保留个人版**。朵朵自己仍在使用原项目（数学/英语/对话）。方案 A 会彻底改造原项目，导致个人版功能受损。方案 B 让两个版本独立存在，各取所需。

5. **老师的使用场景更特化**。课外班的需求（班级管理、共用资料、错题整合）与个人学习工具的需求差异很大。一个为班级教学设计的系统，比一个「个人工具 + 班级补丁」的系统体验更好。

### 实施策略

```
阶段 1：骨架搭建（4-6 小时）
├── 复制核心工具层（ai_client、file_lock、pricing、paths）
├── 设计新数据模型（班级/用户/资料/错题/用量）
├── 搭建 FastAPI 骨架 + 新路由
└── 搭建 Vue 3 骨架 + 新路由

阶段 2：核心功能（8-10 小时）
├── 认证 + 权限体系（三级角色）
├── 班级 CRUD + 学生管理
├── 英语上传 + AI 分类（资料存入班级）
└── 英语解题（强制 Kimi Code，SSE 流式）

阶段 3：高级功能（6-8 小时）
├── 班级资料库（标签、检索、预览）
├── 个人错题本 + 班级错题聚合
├── 用量统计（个人 + 班级汇总）
└── 管理员面板（班级管理 + 全局监控）

阶段 4：打磨（4-6 小时）
├── 前端 UI 优化（液态玻璃风格延续）
├── 权限边界测试（学生/班级管理员/总管理员）
├── 数据隔离测试（资料共享、错题私有）
└── 部署脚本 + 使用文档
```
